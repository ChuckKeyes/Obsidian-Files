Migration Process

1. Estimate
High‑level sizing, timeline, and cost model.
2. Discover
Inventory apps, data, dependencies, and access paths.
3. Assess
Prioritize workloads; identify risks, controls, and quick wins.
4. Plan
  • Rehost — lift‑and‑shift to VMs behind an HTTPS load balancer
  • Replatform — containerize and deploy to Cloud Run or GKE
  • Refactor — modularize services; adopt managed data & CI/CD
5. Migrate
  • Security & IAM — org policy, least‑privilege roles, secrets
  • Networking — VPCs, NCC/HA‑VPN, Cloud Armor/WAF, LB/CDN
  • Instrumentation — Logging, Monitoring, tracing, SLOs
  • Cost Control — budgets, alerts, tagging, FinOps reviews

Example: Migrate a Web Server to GCP

Option A — Cloud Run (serverless)

  • Containerize the site; deploy to Cloud Run with HTTPS
  • Map keyescloudsolutions.com via managed certs
  • Use DNS CNAME for www; use ALIAS/ANAME or A/AAAA at apex (DNSimple supported)

Option B — Compute Engine (VM)

  • Deploy VM template/MIG behind HTTPS Load Balancer
  • Terminate TLS at the global LB (managed certificate)
  • Point DNS A/AAAA to the LB's global IP

We automate both approaches with Terraform, including DNS (DNSimple), health checks, firewall rules, and CI/CD.
